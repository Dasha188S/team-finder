# TeamFinder

Веб-приложение для поиска команды на pet-проекты. Финальный проект курса
Яндекс Практикума, **Вариант 2** (навыки пользователей и фильтрация по навыкам).

Стек: Django 5.2, PostgreSQL 16, Pillow (генерация аватаров), Gunicorn,
WhiteNoise, Docker Compose.

---

## Что реализовано

### Общая функциональность

- Главная страница `/projects/list/`: карточки проектов, сортировка по дате
  (новые сверху), пагинация по 12 проектов на странице. Корневой `/`
  редиректит на главную.
- Страница проекта `/projects/<id>/`: автор, дата, описание, статус, ссылка
  на GitHub, список участников, действия для авторизованных и владельца
  (присоединиться/отказаться, редактировать, завершить, поделиться ссылкой).
- Создание/редактирование проекта `/projects/create-project/` и
  `/projects/<id>/edit/` через одну Django-форму, валидация ссылки на GitHub
  (должна быть валидной и вести на github.com), статусы `Открыт`/`Закрыт`.
- Завершение проекта `/projects/<id>/complete/` (POST) и переключение
  участия `/projects/<id>/toggle-participate/` (POST) — JSON-ответы для UI.
- Регистрация `/users/register/`, вход `/users/login/`, выход
  `/users/logout/`, смена пароля `/users/change-password/`.
- Профиль пользователя `/users/<id>/` с публичной информацией, навыками
  и проектами; редактирование профиля `/users/edit-profile/`.
  Валидация телефона (формат `8XXXXXXXXXX`/`+7XXXXXXXXXX`, нормализация в
  `+7…`, уникальность) и ссылки на GitHub.
- Список пользователей `/users/list/`: карточки, сортировка по `id`,
  пагинация по 12 профилей, фильтр по навыкам через `?skill=<Название>`.

### Особенности варианта 2

- Модель `Skill` (`users.Skill`).
- На странице пользователя есть блок «Навыки»: список тегов; владелец
  профиля может добавлять и удалять навыки **без перезагрузки страницы**.
- Автодополнение `/users/skills/?q=<префикс>`: первые 10 совпадений по
  алфавиту, реализовано через ORM (`name__istartswith`), без выгрузки всех
  навыков в Python.
- Создание нового навыка прямо из интерфейса
  (`POST /users/<id>/skills/add/` с полем `name`).
- Удаление навыка через `POST /users/<id>/skills/<skill_id>/remove/`;
  сам навык остаётся в БД и может быть привязан к другим пользователям.
- Фильтр по навыкам на странице участников с подсветкой активного тега
  и кнопкой «Сбросить».

### Права доступа

| Роль | Возможности |
| --- | --- |
| Гость | Просмотр проектов, профилей, главной и страниц проекта/пользователя; регистрация, вход. |
| Авторизованный | Всё перечисленное + создание/редактирование/завершение своих проектов, участие в чужих, управление навыками, редактирование профиля и смена пароля. |
| Администратор | Всё перечисленное + полное управление пользователями, проектами и навыками через `/admin/`. |

### Дополнительно (пункты «не обязательно», но реализованы)

- Свой Dockerfile (Python 3.12-slim) и `docker-compose.yml`,
  поднимающий **db и web** одной командой.
- Persistence: `volumes` для PostgreSQL (`postgres_data`),
  пользовательских аватаров (`media_data`) и собранной статики
  (`static_data`).
- CI на GitHub Actions: `flake8` (PEP 8, max-line-length=100) и
  `python manage.py test` (см. `.github/workflows/ci.yml`).
- Автотесты на ключевую логику (`users/tests.py`, `projects/tests.py`).
- Management-команда `python manage.py seed_demo` создаёт суперпользователя,
  5 тестовых пользователей с навыками и 5 проектов с участниками.

## Быстрый запуск через Docker Compose (рекомендуется)

Это **самый простой способ** проверить проект — поднимет PostgreSQL и
Django одной командой и автоматически наполнит БД тестовыми данными.

```bash
git clone <URL-репозитория>
cd team-finder
cp .env_example .env
docker compose up --build
```

После старта:

- Приложение: <http://localhost:8000/>
- Админ-панель: <http://localhost:8000/admin/>

При первом запуске entrypoint:

1. Дожидается готовности PostgreSQL.
2. Применяет миграции (`migrate`).
3. Собирает статику (`collectstatic`).
4. Если переменная окружения `SEED_DEMO=1` (по умолчанию в
   `docker-compose.yml`), создаёт демо-данные.

### Остановить и очистить

```bash
docker compose down              # остановить
docker compose down -v           # остановить и стереть volumes
```

## Локальный запуск без Docker (опционально)

```bash
# 1. Виртуальное окружение
python3 -m venv venv
source venv/bin/activate         # для Windows: venv\Scripts\activate

# 2. Зависимости
pip install -r requirements.txt

# 3. Поднять только PostgreSQL в контейнере
docker compose up -d db
cp .env_example .env
# и заменить в .env POSTGRES_HOST=db -> POSTGRES_HOST=localhost

# 4. Миграции, демо-данные, сервер
python manage.py migrate
python manage.py seed_demo
python manage.py runserver
```

Создать своего суперпользователя без `seed_demo`:

```bash
python manage.py createsuperuser
```

## Тесты и линтер

```bash
# Тесты (используют SQLite-бэкенд автоматически, БД не нужна)
python manage.py test

# PEP 8 (max-line-length = 100)
flake8 .
```

Тесты покрывают: модели, регистрацию/вход/выход, валидацию профиля,
автодополнение навыков, добавление/удаление навыков, права доступа на
редактирование/завершение проекта, переключение участия, пагинацию.

CI запускает оба шага на GitHub Actions для каждого пуша и PR.

## Особенности и принятые решения

- **Логин по email**. Стандартная модель `User` Django заменена на
  `users.User` с `USERNAME_FIELD = "email"` и кастомным `UserManager`.
- **Аватар обязателен**. При создании пользователя автоматически
  генерируется PNG с первой буквой имени на однотонном тёмном фоне
  (палитра подобрана так, чтобы белая буква была хорошо читаемой).
  Реализовано в `users.utils.generate_avatar` через Pillow и вызывается
  из `User.save()`. Используемый шрифт — `DejaVu Sans Bold`
  (устанавливается из пакета `fonts-dejavu-core` в Docker-образе),
  с фолбэком на встроенный шрифт Pillow.
- **Регистрация → вход**. По чек-листу ревьюера после регистрации
  пользователь попадает на `/users/login/` (а не сразу авторизуется).
- **Тесты используют SQLite** через `if "test" in sys.argv` в
  `settings.py`, чтобы их можно было гонять локально и в CI без
  поднятого PostgreSQL.
- **WhiteNoise** обслуживает статику в production-сборке, чтобы
  `gunicorn` отдавал её без отдельного nginx.
- Пагинация и навигационные ссылки одинаково корректно работают и без
  завершающего слэша, и с ним (дублирующие маршруты в `users/urls.py` и
  `projects/urls.py`) — это нужно потому что часть существующих
  HTML-шаблонов из стартового набора ходит по ссылкам без слэша
  (`/users/list`, `/projects/{id}/edit` и т. п.).
- Стартовый шаблон `templates_var2/users/participants.html` содержал
  фильтр из варианта 1 (4 заранее заданных критерия) и не имел фильтра
  по навыкам. Шаблон переписан для соответствия требованиям варианта 2:
  панель тегов из всех навыков, подсветка активного, кнопка «Сбросить».
- Стартовый шаблон `templates_var2/users/user-details.html` не содержал
  блока навыков — добавлен по аналогии с шаблоном проекта из варианта 3.
- Шаблон `templates_var2/projects/favorite_projects.html` и подключение
  `project-fav-toggle.js` удалены, потому что вариант 2 не предполагает
  избранного.
